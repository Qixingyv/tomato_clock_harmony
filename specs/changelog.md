### v1.1 (2026-03-18)

**变更类型**：修改

**变更内容**：
- 为待办模式添加"已取消"（Cancelled）状态
- 状态码使用 5，与计划模式的状态码区分
- 已取消任务移入历史记录，可查看和恢复
- 更新状态流转图，添加取消分支
- 更新筛选条件，添加"已取消"选项

**理由**：
- 用户可能创建任务后发现不需要，需要取消功能
- 已取消状态提供可恢复性，比直接删除更友好
- 与计划模式的"已归档"状态形成对应

**影响范围**：
- `functional/task-modes.md` - 待办模式状态定义
- `functional/task-management.md` - 状态流转、筛选条件
- `technical/data-model.md` - TaskStatus枚举、状态字段说明

**数据模型变更**：
```typescript
// 新增状态
export enum TaskStatus {
  Working = 1,    // 进行中
  Completed = 2,  // 已完成
  Archived = 3,   // 已归档（计划模式）
  Expired = 4,    // 已过期（计划模式）
  Cancelled = 5   // 已取消（待办模式）新增
}
```

### v1.3 (2026-03-18)

**变更类型**：修改

**变更内容**：
- 确认计划模式采用**方案A**：用户设置（每天番茄数 + 坚持天数）
- 系统自动计算累计总目标：`total = daily × days`
- 简化用户心智模型："每天做多少" + "坚持多久"
- 任务卡片显示优化：突出今日进度，累计数据作为辅助展示

**理由**：
- 番茄钟是习惯养成工具，用户更关心"今天的我有没有坚持"
- "每天5个"比"100天500个"更具体、更可执行
- 符合计划模式的定义：每日重复的习惯

**数据模型变更**：
```typescript
// 用户输入
daily_target_tomato_count: number;  // 每天目标
deadline: Date;                      // 截止日期（计算天数）

// 系统计算
total_target_tomato_count: number;  // = daily × days
```

**影响范围**：
- `functional/task-modes.md` - 重写计划模式定义，添加用户心智模型
- `functional/task-management.md` - 更新创建表单、任务卡片显示
- `technical/data-model.md` - 更新字段说明

### v1.4 (2026-03-18)

**变更类型**：修改

**变更内容**：
- 整合待办模式和计划模式的"取消"操作
- 计划模式新增"已取消（5）"状态
- 统一取消操作的用户体验：两种模式都使用"取消"按钮
- 已取消任务可恢复，移入历史记录
- 更新状态流转图，添加取消分支
- 更新筛选条件，添加"已归档"选项

**理由**：
- 用户操作统一：两种模式的删除/取消行为应该一致
- 语义清晰：取消（中途放弃）与归档（完成后归档）明确区分
- 代码简洁：统一的取消逻辑，减少特殊处理
- 可恢复性：取消比直接删除更友好，用户可以反悔

**影响范围**：
- `functional/task-modes.md` - 计划模式状态定义、行为规则、状态流转
- `functional/task-management.md` - 编辑操作、状态定义、状态流转、筛选条件
- `technical/data-model.md` - 状态字段说明

**数据模型变更**：
```typescript
// 计划模式新增状态
export enum TaskStatus {
  Working = 1,    // 进行中（通用）
  Completed = 2,  // 已完成（通用）
  Archived = 3,   // 已归档（计划模式专属）
  Expired = 4,    // 已过期（计划模式专属）
  Cancelled = 5   // 已取消（通用）✨ 扩展到两种模式
}
```

**统一后的状态流转：**
```
待办模式：  [创建] → 进行中(1) → 已完成(2)
                     ↓
                已取消(5)

计划模式：  [创建] → 进行中(1) → 已完成(2) → 已归档(3)
                     ↓            ↑
                已取消(5)    用户操作
                     ↓
                可恢复
```

### v1.5 (2026-03-18)

**变更类型**：修改

**变更内容**：
- 移除"已归档（3）"状态
- "已完成"和"已归档"合并为单一"已完成（2）"状态
- 简化状态系统，减少一个状态码
- 更新筛选条件，移除"已归档"选项
- 历史任务通过时间筛选管理

**理由**：
- MVP应聚焦核心功能，归档不是必需的
- 简化状态系统，降低开发和维护成本
- 用户可通过时间筛选（今日/本周/本月/更早）管理历史任务
- 如后续需要，可在v2.0添加深度归档功能

**影响范围**：
- `functional/task-modes.md` - 计划模式状态定义、状态流转
- `functional/task-management.md` - 状态定义、状态流转、筛选条件
- `technical/data-model.md` - TaskStatus枚举、状态字段说明

**数据模型变更**：
```typescript
// 移除前
export enum TaskStatus {
  Working = 1,    // 进行中
  Completed = 2,  // 已完成
  Archived = 3,   // 已归档 ❌ 移除
  Expired = 4,    // 已过期
  Cancelled = 5   // 已取消
}

// 移除后
export enum TaskStatus {
  Working = 1,    // 进行中（通用）
```

**简化后的状态系统**：
```
通用状态（两种模式）：
  1 = 进行中
  2 = 已完成
  5 = 已取消

计划模式专属：
  4 = 已过期
```

### v1.6 (2026-03-18)

**变更类型**：修改

**变更内容**：
- 确认历史记录功能集成到统计页面
- 统计页面新增子Tab导航：[概览] [历史记录]
- 概览Tab：展示统计数据（今日/本周/本月）
- 历史记录Tab：展示已完成/已取消/已过期任务列表
- 添加状态筛选和时间筛选功能

- 减少主导航Tab数量，保持3个核心Tab：任务列表、统计、我的
- 用户使用场景自然连贯：完成番茄钟 → 查看统计 → 切换历史看详情
- MVP开发成本更低，后续可独立出来（v2.0）

**影响范围**：
- `functional/task-management.md` - 更新历史记录查看入口和详细设计
- `functional/statistics.md` - 添加子Tab导航和历史记录Tab布局

### v1.7 (2026-03-19)

**变更类型**：新增 + 延期

**变更内容**：

**新增功能：**
- MVP阶段新增用户认证系统
- 支持微信一键登录
- 支持华为账号登录
- 实现Token管理（access_token + refresh_token）
- 实现自动登录功能
- 实现登出功能

**延期功能：**
- 统计功能整体延期到v1.1版本
- 历史记录功能延期到v1.1版本

**理由：**

新增用户认证的理由：
- 为后续数据同步做准备
- 提升用户数据安全性
- 支持跨设备使用体验
- 微信和华为账号是鸿蒙生态的主要登录方式
- 减少用户注册门槛，一键登录提升转化率

延期统计功能的理由：
- 聚焦MVP核心价值：番茄钟计时和任务管理
- 统计功能需要大量数据积累才有意义
- 缩短MVP开发周期，快速验证核心功能
- 用户认证为后续统计功能提供数据基础

**影响范围**：
- `functional/user-auth.md` - 新增用户认证功能文档
- `functional/overview.md` - 更新产品概述
- `technical/plan.md` - 更新技术实现方案

- `technical/data-model.md` - 新增用户相关数据模型

## 决策记录

### 决策1：任务模式设计

**日期**：2026-03-18

**决策**：
- 采用待办模式（一次性任务）和计划模式（每日重复任务）
- 计划模式支持双重目标制（每日目标 + 累计总目标）

**理由**：
- 满足用户不同的时间管理需求
- 双重目标制提供更灵活的目标设定

**影响**：
- 数据模型需要支持两种模式的字段
- 计划模式需要每日重置机制

---

### 决策2：计时器功能范围

**日期**：2026-03-18

**决策**：
- 支持专注模式和休息模式
- 支持暂停、继续、提前结束、跳过休息
- MVP版本不处理应用被杀死后的恢复

**理由**：
- 提供完整的计时器体验
- MVP聚焦核心功能，异常恢复放到后续版本

**影响**：
- 计时器组件需要处理多种状态流转
- 数据持久化可放到v2.0

---

### 决策3：统计功能范围

**日期**：2026-03-18

**决策**：
- MVP版本只提供基础统计（今日/本周/本月）
- 统计指标：完成番茄数、专注时长、任务完成率
- 暂不实现图表可视化

**理由**：
- 满足基本的数据反馈需求
- 图表可视化可作为增强功能

**影响**：
- 统计页面以数字卡片为主
- 为后续图表功能预留扩展空间

---

### 决策4：时长配置策略

**日期**：2026-03-18

**决策**：
- 采用"全局默认 + 可选覆盖"策略
- 在设置中配置默认时长
- 每个任务可单独设置时长

**理由**：
- 平衡便捷性和灵活性
- 大部分任务使用默认值，特殊任务可自定义

**影响**：
- 需要设置页面
- 任务创建时自动填充默认值

## 争议问题

### ~~问题1：计划模式累计总目标是否必要？~~ ✅ 已解决

**背景**：
- 当前设计支持累计总目标（可选）
- 增加了数据模型复杂度

**正方观点**：
- 用户可能希望设定长期目标（如"100天学习计划"）
- 提供更明确的目标感

**反方观点**：
- 计划模式主要是每日重复，累计总目标不是必需的
- 可以用截止日期代替

**✅ 决策（v1.3）**：
- 采用**方案A**：用户设置"每天番茄数"和"坚持天数"
- 系统自动计算累计总目标作为后台字段
- 保留数据结构，主要用于统计展示
- 不在创建界面显示累计目标输入

**相关决策记录**：参见 决策6
